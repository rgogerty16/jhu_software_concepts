# Module 7 — Cloud Computing (AWS)

Two deliverables:

1. **Part 1 — S3 → SageMaker (boto3):** pull the Grad Café dataset from a private
   S3 bucket into a SageMaker notebook and save it locally as
   `applicant_data_SM.json`.
2. **Part 2 — Deploy Module 6 to EC2:** run the Module 6 Docker Compose stack
   (Flask **web** + **worker** + **Postgres** + **RabbitMQ**) on an EC2 instance.

> **AWS resources were STOPPED after completion** (SageMaker notebook instance and
> EC2 instance are stopped, not terminated — they are reused in Module 8). No AWS
> credentials, SSH keys, or `.env` files are committed.

## Part 1 — Run the SageMaker notebook

**On the SageMaker notebook instance** (`s3-to-sagemaker-grad-cafe-pipeline`,
`ml.t2.medium`, with an IAM role scoped to the `grad-cafe` bucket):

1. Upload `grad-cafe-pipeline.ipynb`, `s3_fetch.py`, and `requirements.txt` into
   the Jupyter working directory (same folder — the notebook does `import s3_fetch`).
2. In a cell or terminal: `pip install -r requirements.txt`.
3. **Run All.** The notebook:
   - downloads `applicant_data.json` from the `grad-cafe` S3 bucket, and
   - **saves it locally as `applicant_data_SM.json`** in the notebook's working
     directory (printed as `Saved to: applicant_data_SM.json`), then
   - prints a summary (record count + first-record fields) as proof.
4. Download the executed notebook (with outputs) back into `module_7/` for submission.

**How the download works / credentials:** all boto3 logic is in `s3_fetch.py`.
`make_s3_client()` builds an S3 client from **boto3's default provider chain** — on
SageMaker that is the notebook instance's attached **IAM execution role**, so **no
keys are hard-coded** anywhere. `download_dataset()` calls
`client.download_file(bucket, key, output_path)`. Bucket/key/output/region are
overridable via env vars (`S3_BUCKET`, `S3_KEY`, `OUTPUT_FILE`, `AWS_REGION`; see
`.env.example`) so `grad-cafe-<initials>` works too.

> `applicant_data.json` / `applicant_data_SM.json` are git-ignored (15–18 MB,
> regenerated by running the notebook).

## Part 2 — Deploy to EC2

Full runbook: **[`ec2/EC2_DEPLOYMENT.md`](ec2/EC2_DEPLOYMENT.md)**. In short:

1. Launch **Ubuntu 22.04 `t3.micro`**; key pair for SSH; security group opens **22
   and 8080 to your IP only** (5432 and 15672 stay private).
2. SSH in; install Docker + the Compose plugin
   (`sudo apt-get install -y docker.io docker-compose-plugin`; add your user to the
   `docker` group); verify `docker --version` / `docker compose version`.
3. `scp` `ec2/docker-compose.ec2.yml`, `ec2/init.sql`, your `.env`, and
   `module_6/src/data/applicant_data.json` (→ `~/app/data/`) up to the instance.
4. Deploy:
   ```bash
   docker compose -f docker-compose.ec2.yml --env-file .env up -d
   docker compose -f docker-compose.ec2.yml ps
   ```
5. Verify: browse `http://<EC2_PUBLIC_IPV4>:8080`, then click **Pull Data** to prove
   the **web → RabbitMQ → worker → Postgres** background path works in the cloud.

The `web`/`worker` images are pulled from Docker Hub
(`rgogerty/module_6:web-v1` / `worker-v1`); `db` is `postgres:16` with a persistent
`pgdata` volume; `rabbitmq` is `3.13-management`. `EC2_DEPLOYMENT.md` also lists the
`docker build … && docker push` commands in case the images need publishing first.

## Security notes

- No AWS access keys, SSH private keys (`*.pem`), IAM `accessKeys*.csv`, or `.env`
  files are committed (see `.gitignore`).
- The SageMaker notebook uses its IAM **execution role**, not embedded keys.
- On EC2, Postgres and RabbitMQ are never exposed publicly; only port 8080 is open,
  restricted to a single IP in the security group.
